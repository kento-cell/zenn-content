## 【絶句】AIコーディングが止まらない時代に。WebからパラメトリックCADを動かすアーキテクチャ設計論

正直、最近のエンジニアリングの進化スピードは「狂気」レベルですよね。(^_^)

特にAIコーディングツールのおかげで、機能的なプロトタイプを作るのは以前と比べてマジで圧倒的に楽になりました。でも、ブっちゃけ、その手軽さの裏側には、「設計者が本当に求めるインタラクティブなフィードバックループ」という、目に見えない壁が立ちはだかっているのが現状なんです。

みなさん、「AIにコードを書いてもらうのは楽しいけど、なんか物足りない…」と感じたこと、ありませんか？ 私はこの「あと一歩のモヤモヤ」こそが、今のWebエンジニアリングにおける最大の課題だと考えています。

この記事では、単なるツールの紹介ではなく、パラメトリックな物理シミュレーションやCAD設計のような複雑なバックエンドロジックを、いかにして**ブラウザ上で高性能かつインタラクティブに再現するか**という、アーキテクチャの視点から深掘りしていきます。これは、Webエンジニアなら「これ、次のSaaS案件で使えるぞ」と感じてもらえるはずです！

## 1. プロトタイピングの限界：「静的なコード生成」と「動的パラメータ調整」のギャップ

最新のAIは驚異的です。テキストプロンプトを出すだけで、動くWebアプリケーションやロジックが生成されていくのは目覚ましいものがありますよね。しかし、この進化の波に乗って「設計」という領域に踏み込んでみると、ある種の構造的な限界が見えてきます。

そもそもCAD設計とは、「静的な形物を作るプロセス」ではなく、「**パラメータ（変数）を操作しながら最適な形状へ収束させていく検証プロセス**」そのものなんです。

例えば、3Dプリンタ用のロボットケースを考えてみてください。AIに「このサイズの箱を作って」とコードで生成させたとしても、設計者は必ず以下のような微調整を行いますよね。

* 「あと0.5mmだけ外側に広げたい。」
* 「この穴の開け位置をZ軸方向に2mmずらして、配線ルートを最適化したい。」
* 「クライアントが『やっぱりここはスライドさせて見ながら決めたい』って言う。」

このような「**パラメータの微調整による動的な検証（Exploration）**」のフェーズは、従来のコード生成や単なるビューアでは対応しきれません。

> "AIコーディングツールにコードを書いてもらう体験は、かなり実用的になってきました。一方で、3Dプリンタ用の部品やロボットのケースを作るようなCAD設計では、まだ少しもどかしさがあります。 CadQueryのようなコードベースのCADを使えば、AIに「幅を何mmにして、ここに穴をあけて」と頼めます。ただ、実際の設計では、最終的に「あと0.5mm広げたい」「この穴は少し上にずらしたい」「スライダーで見ながら追い込みたい」という場面が出てきます。"
>
> 出典: []/Zenn トレンド. "AIと一緒にCAD設計できるソフト「cad-coworker」"
> https://zenn.dev/karaage0703/articles/47299d42de9607

この引用が示している「もどかしさ」こそ、我々Webエンジニアが今解決すべき技術的な課題の本質なんです。**ローカル環境で動くPythonロジックを、ブラウザのインタラクティブなUI（スライダーや入力フォーム）と結びつけるパイプライン**が必要不可欠になっているんですよね。(^^)

## 2. 「Webファースト」パラメトリックCAD実現のためのアーキテクチャ分析

どうすれば、ローカル環境で動くPythonベースの複雑なロジックをブラウザ上で快適に実行できるのか？ これが最大の技術課題です。単なるファイルアップロードやAPIコールでは、リアルタイム性が求められる「追い込み」の感覚は再現できません。

私はこの問題を解決するためには、「計算処理自体をWeb側に極力近づけるか」「バックエンドとフロントエンド間の通信オーバーヘッドを最小化する設計」が必須だと考えています。

### 2.1. 技術選択肢の比較検討：どのロジックをどこで動かすべきか？

技術スタックを考える際、まず「計算負荷（Computation Load）」と「レイテンシ（Latency）」のトレードオフからアプローチする必要があります。

| アプローチ | メリット | デメリット | 最適なユースケース |
| :--- | :--- | :--- | :--- |
| **A. WebAssembly (WASM) によるブラウザ実行** | ネットワーク依存が低く、リアルタイム性が極めて高い。PythonロジックをWeb側で動かせる可能性が高い。 | パッケージングや言語の橋渡し（FFI）が非常に複雑。デバッグ難易度が高い。 | 小規模な幾何計算、即時フィードバックが必要なシミュレーション。 |
| **B. WebSocket + 強力なバックエンド (Python/FastAPI)** | 既存の強力なバックエンドロジックをそのまま利用できる。状態管理が容易。 | ネットワークレイテンシの影響を受けやすい。高頻度のステート変更には負荷がかかる。 | パラメータ調整後のレンダリング（静的結果の表示）、複雑なデータ処理。 |
| **C. WebGL/Three.js + バックエンドAPI** | フロントエンドで描画に特化し、UIは高速。計算ロジックと描画を分離できる。 | 幾何学計算そのものをフロントで行うのは非常に難しい（バグが入りやすい）。 | パラメータ変化による視覚的な変更（色、拡大縮小など）のフィードバック。 |

**筆者の見解として、最も現実的かつ強力なアプローチは「B. WebSocket + 強力なバックエンド」を主軸としつつ、「A. WASM」でコアな幾何計算部分の一部を高速化するハイブリッド構成です。**

なぜなら、CADのロジックは一般的に巨大であり、これを完全にブラウザに移植するのは現実的ではないからです。まずは「パラメータを受け取り→計算し直す→結果をJSON形式で返す」というパイプラインが最優先となります。

### 2.2. アーキテクチャ図：State-Driven Parametric Pipeline

このハイブリッドなアプローチに基づいたシステムフローは、以下のような構造になります。Mermaid記法を使って可視化してみますね。

```mermaid
graph TD
    A[ユーザー入力: Web UI （スライダー/フォーム）] -->|パラメータ変更(P1, P2...)| B{WebSocket接続};
    B --> C[バックエンドAPI層 （FastAPI / Python）];
    C --> D{Parametric Core Logic (CadQuery + WASM)};
    D --> E[ジオメトリ計算実行 （Numpy/SciPyなど）];
    E --> F[結果の正規化/軽量JSON出力];
    F -->|更新されたモデルデータ(Mesh/JSON)| B;
    B --> G[フロントエンドレンダリング層 （Three.js / React Three Fiber）];
    G --> H[画面描画 （ユーザーへフィードバック）];

    subgraph 処理の流れ
        C & D & E
    end
```

この図から読み取れるのは、「**状態（State）に基づいた、イベント駆動型のデータフロー**」こそが肝だということです。ユーザーがスライダーを動かす→（イベント発生）→WebSocketでパラメータ送信→（バックエンド）→再計算ロジック実行→（結果のJSON化）→フロントで即時描画というサイクルを極限まで高速にすることが求められます。

### 2.3. 実装コード例：Websocketによる状態同期の実装骨子

**【バックエンド (Python/FastAPI)**
```python
from fastapi import FastAPI, WebSocket
import json

app = FastAPI()

def run_parametric_cad_logic(params: dict) -> dict:
    """
    パラメータを受け取り、新しい3Dモデルの状態を計算するシミュレーション。
    ここでは、簡単な体積と穴の位置の変化のみを返す例とする。
    """
    try:
        width = float(params.get("width", 10))
        height = float(params.get("height", 5))
        hole_z = float(params.get("hole_z", 2))

        ## 計算ロジックの実行 (この部分が重い計算なので、最適化が必要)
        volume = width * height * 10 - hole_z * 5
        
        return {
            "success": True,
            "new_state": {
                "width": width,
                "height": height,
                "hole_z": hole_z,
                "volume": round(volume, 2) # 計算結果をJSON化して送信
            }
        }
    except Exception as e:
        return {"success": False, "error": str(e)}

@app.websocket("/ws/cad-update")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()
    try:
        while True:
            ## クライアントからパラメータデータを受け取る
            data = await websocket.receive_json() 
            params = data.get("parameters", {})

            print(f"Received parameters: {params}")

            ## ロジック実行と結果取得
            result = run_parametric_cad_logic(params)

            ## 結果をクライアントに送信（これが即時フィードバックの鍵）
            await websocket.send_json({"status": "completed", "data": result})

    except Exception as e:
        print(f"WebSocket connection closed: {e}")
```

**【フロントエンド (React / TypeScript)**
```typescript
// 擬似的なTypeScriptコード（概念理解用）

import React, { useState, useEffect } from 'react';
import * as ws from 'websocket-client'; // 仮のWSクライアントライブラリ

const ParametricCADView: React.FC = () => {
    const [params, setParams] = useState({ width: 10, height: 5, hole_z: 2 });
    const [modelState, setModelState] = useState(null);

    useEffect(() => {
        // WebSocket接続確立
        const wsConnection = new ws('ws://localhost:8000/ws/cad-update');

        wsConnection.onmessage = (event) => {
            const data = JSON.parse(event.data);
            if (data.status === "completed") {
                // 計算結果を受け取り、モデルの状態を更新する
                setModelState(data.data.new_state); 
            }
        };

        // クリーンアップ処理
        return () => wsConnection.close();
    }, []);

    // パラメータが変更されるたびにバックエンドに送信するハンドラー
    const handleParamChange = (paramName: string, value: number) => {
        setParams(prev => ({ ...prev, [paramName]: value }));
        
        // 即時フィードバックのため、パラメータ変更時にwsを叩く（デバウンス処理必須）
        wsConnection.send(JSON.stringify({ parameters: params })); 
    };

    return (
        <div>
            {/* スライダーUIの描画 */}
            <input type="range" value={params.width} onChange={(e) => handleParamChange('width', parseFloat(e.target.value))} />
            {/* ... 他のパラメータも同様に続く */}

            {/* 3Dモデル表示エリア (Three.jsなどで描画) */}
            <div className="model-viewer">
                {modelState ? `現在の状態: 幅=${modelState.width}, 体積=${modelState.volume}` : "計算中..."}
            </div>
        </div>
    );
};
```

この設計の肝は、**パラメータの変化（イベント）をトリガーとして、バックエンドに「再計算要求」を送りつけ、その結果によってフロントエンドの状態が更新される**という、明確なデータフローを持つ点です。これがWebエンジニアリングにおける高性能インタラクティブアプリケーションの基本形だと捉えてほしいです。(￣▽￣)

## 3. パラメトリック設計から学べる「UI/UX」の本質的課題

この技術的な深掘りを通じて、私は単なるアーキテクチャの話に留まらない、「Webエンジニアがデザイナー視点を持つべき本質的な課題」を見つけました。それが**「ユーザーの思考プロセスをインターフェースとして取り込むこと」**です。

CAD設計におけるパラメトリックな調整は、まさにこの考え方に基づいています。単なるボタンクリックではなく、「もしこれを変えたら、どうなるか？」という問いに対する即時的なフィードバックが必須なんですよね。

### 3.1. UIの役割を「コントロールパネル」として再定義する

従来のWebアプリケーションでは、UIは「操作（Action）」を提供することが主眼でした。しかし、パラメトリックな設計ツールにおいては、UIそのものが**「シミュレーション空間への介入ポイント＝コントロールパネル」**としての役割を果たします。

つまり、スライダーや入力ボックスをただ置くだけでなく、「このスライダーの値が変化すると、ジオメトリのどこに影響するか？」という因果関係（Causality）をユーザーに明確に伝える必要があります。

| 従来のWeb UI | パラメトリックな設計ツールにおけるUI |
| :--- | :--- |
| **目的:** 特定のアクションを実行させる (例: 購入ボタン) | **目的:** 変数を操作し、結果の予測・検証を行う (例: 幅を広げる) |
| **フィードバック:** アクション完了（成功メッセージ） | **フィードバック:** ジオメトリの変化（リアルタイムな形状変化） |
| **設計原則:** 直感的な導線とゴール設定 | **設計原則:** 変数間の依存関係の可視化と操作性 |

### 3.2. 計算コストを隠蔽するUX設計：デバウンスとアシンクロナス処理

前述したように、CADロジックは計算負荷が非常に高いです。もしユーザーがスライダーをマウスで素早く動かした場合（＝高頻度なイベント）、毎回バックエンドにフルリクエストを送るのは、システムにとって大きな負担になりますし、レイテンシも発生します。

ここで重要になるのが「**デバウンス (Debounce)**」と「**アシンクロナス処理 (Asynchronous Processing)**」の適切な組み合わせです。

1. **Debounce:** ユーザーが操作を止め、一定時間（例：200ms）経過してから初めて再計算要求を送る仕組み。これにより、無駄なAPIコールを防ぎます。
2. **アシンクロナス処理:** 計算が重い場合、即座に結果を待つのではなく、「現在計算中です...」というローディングステートを適切に表示しつつ、バックグラウンドで処理を進める設計が必要です。

この二つの技術的工夫こそが、Webエンジニアリングのスキルセットを「単なるロジックの実装」から「**高性能なユーザー体験（UX）の提供**」へと引き上げる境界線だと筆者は考えています。(´・ω・`)

## 4. 実務への応用示唆：このパラダイムはどこに使えるか？

ここまで技術的なアーキテクチャと、それに伴うUXの本質について深掘りしてきましたが、「結局、これをどうビジネスに応用すればいいんだ？」というのが最も気になる点だと思います。w ｗ

私は、この「パラメータによる動的シミュレーション」の考え方は、CADという物理分野に限定されるものではないと考えています。むしろ、**ユーザーが変数を操作することで最適な答えを導き出す必要があるすべてのSaaS領域**に応用できるはずです。

### 4.1. 応用可能性の高い3つのユースケース

| ユースケース | 変数（パラメータ） | シミュレーションロジック | 期待される価値 |
| :--- | :--- | :--- | :--- |
| **LP・ウェブサイト最適化** | 画像サイズ、CTAボタンの配置、キャッチコピーのトーン＆マナー | クリック予測率（CTR）、離脱率シミュレーション | A/Bテストを事前に仮想環境で実施し、最適なレイアウトを決定できる。 |
| **金融商品設計** | 投資期間、リスク許容度、目標リターン率 | リスク・リターン曲線計算 (モンテカルロシミュレーション) | 「このパラメータを変更すると、年利がどう変動するか」を動的に可視化する。 |
| **業務フロー自動設計** | ステップ数、承認者の数、工数配分、システム連携APIの有無 | 全体リードタイム計算 (PERT分析) | 複数のボトルネックをシミュレーションし、「最も効率的なプロセス改善案」を提案する。 |

### 4.2. パイプライン開発のためのワークフロー整理

もしあなたが次にこの種の「動的シミュレーションSaaS」を設計する場合、以下のステップを踏むことを強く推奨します。これは単なる技術選定ではなく、プロジェクトの成功確率を上げるためのプロセスです。

1. **ロジック抽出とモデリング（非Webフェーズ）:** 最初に、物理的な計算やビジネスルールを完全に分離し、純粋な関数として定義する。この段階で「入力パラメータ」と「出力結果」のみに焦点を当てる。（例：Pythonのクラス構造化）。
2. **データ層の設計:** 計算に必要なすべての定数、単位、依存関係（Constraint）を明確に洗い出す。これにより、UI側に表示すべきパラメータが確定します。
3. **APIゲートウェイ構築（Webフェーズ）：** 抽出したロジックをラップする形でバックエンドAPIを作成し、「パラメータを受け取る→計算を実行する」という最小限の機能のみを実装する。（まずはWebSocketは不要で良い）
4. **フロントエンドUIのMVP開発:** 計算結果をただ表示させるだけでなく、ユーザーが「変数をいじっている感覚」を持てるよう、スライダーとプログレスバーなど、インタラクティブな要素から作り始める。

このワークフローを守ることで、「なんか技術的に難しい」という漠然とした不安を、「どのレイヤーで、どんなデータを渡せばいいか？」という具体的な設計課題に落とし込むことができるはずです。(╹◡╹)

## 結論：エンジニアは「計算の動的表現力」を提供すべき時代へ

ここまで読んでいただきありがとうございます。ぶっちゃけ、AIがコードを生成してくれる時代だからこそ、我々Webエンジニアが真に価値を発揮できる領域は、「**単なるコード記述能力ではなく、複雑なロジックをいかにインタラクティブかつ直感的に表現し、ユーザーの思考プロセスを導き出すか**」という部分にあると確信しています。

目の前の「パラメータ調整」という課題から学べるのは、技術スタック以上に**システム全体の設計思想（アーキテクチャ）が重要だ**ということです。計算ロジックを分割し、WebSocketやデバウンスといった通信・状態管理の仕組みを駆使することで、巨大なバックエンド処理をまるでネイティブアプリのようにサクサク動くWeb体験に変貌させることができます。

次は、この「インタラクティブなシミュレーションパイプライン」という視点を持ち、あなたが携わる次のSaaS案件で、「ユーザーが変数を操作して最適な答えを見つけるためのインターフェース」の設計から取り組んでみてください。その挑戦こそが、今のWebエンジニアリングにおける最もホットな領域だと筆者は強く感じています！
