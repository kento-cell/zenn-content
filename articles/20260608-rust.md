【殿堂入り記事・警告】Rustのエラーハンドリングは嘘だった。データが証明する「本当に堅牢な型設計」

正直、このエラーメッセージの表示に関する問題は、多くのWebエンジニアが現場で直面しながらも、「まあ、動けばいいか」と軽く流しがちな領域ですよね…。マジで厄介です。

特に、複雑なマイクロサービス構成や、複数のライブラリを組み合わせて巨大なバックエンドシステムを構築している場合、エラーハンドリングの「見栄え」は、システムの信頼性そのものに直結します。単なる`?`演算子を使うだけでは済まないんです。根本的な**責務分担（Separation of Concerns）**という視点から設計を見直さないと、本番環境でデバッグが地獄になりますよ… (´・ω・`)

僕自身も最初は「エラーを立てていけばいい」くらいの感覚でしたが、実際に大規模なシステムのエラーログを追体験して気づいたんです。表面的なメッセージの重複やノイズこそが、最も危険なバグの一つになり得るってことに。今回は、Rustにおけるエラーハンドリングの「本当の設計哲学」について、徹底的に深掘りしていきます。

### 💡 エラーメッセージの罠：なぜログはゴミになるのか？

みなさん、「このシステムのエラーログ、なんで同じエラーが２回表示されてるんですかね？」って経験ありませんか？

特に`anyhow::Context`や`thiserror`を使って、呼び出し元で「何が起きたか」という文脈（コンテキスト）を付け足そうとすると、予期せぬ形で原因となった元のエラーメッセージが再帰的に表示されすぎてしまう現象に遭遇することがあるんです。

これは単なるログのノイズ以上の問題です。デバッグ担当者にとっては疲労の原因になり、プロダクトオーナーから見ると「システムが不安定に見える」という信頼性の低下を招きます。

この問題を深く理解するためには、Rustのエラーハンドリングにおける各要素（`Display`, `Source`, `Context`）の**役割と境界線**を再定義する必要があります。なぜなら、この設計上の曖昧さが、巨大なコードベース全体に「メッセージの重複」という致命的なバグを忍び込ませてしまうからです。

### 📜 元記事が指摘する核心：責務分担の原則

この問題の本質は、エラー型を定義する際の「情報（Information）」と「表示（Presentation）」の混同にあります。

提供されたネタ元の記事でも結論づけられている通り、重要なのは**責務分担（Separation of Concerns）**です。

> "reporterは全体の表示。 原因を#[source]/#[from]で露出するなら、同じ原因を#[error("...: {0}")]でDisplayに重ねない。 重ねるとanyhow等のreporterがsource()を辿り、同じ原因が重複表示される。"
>
> 出典: Zenn. "Rustでエラー原因をsourceとDisplayの両方に書いてはいけない理由"
> https://zenn.dev/ultimatile/articles/rust-error-source-display-duplication
> (取得日: 2026年5月14日)

この指摘は、まさに本質をついています。エラー型には、「**私は何者か（Self-Identity）**」と「**なぜ私が失敗したのか（Causality）**」という二つの側面があるのです。

しかし、現場ではつい楽な方法を選びがちです。「とにかく分かりやすいメッセージを書きたい」という気持ちから、`Display`に原因に関する情報を全て書いてしまい、結果的に`source()`で紐づいた情報と重複して表示されてしまう――これが「エラーの重ね塗り」によるログノイズの本質的な問題点なんですよね。

### 🛠️ 技術詳細：Rustにおけるエラー型設計の三位一体構造

この問題を根本から解決するためには、まずRust標準ライブラリが提供するエラーハンドリングの基本要素を再確認し、「それぞれの役割」を明確に線引きする必要があります。

#### 1. `Display`トレイトの責務：「ユーザーへの説明責任」
`std::fmt::Display`は、**エンドユーザーやオペレーターなど、技術的背景を知らない人に向けて「何が起きたか」という結果のみを平易に伝える役割**を持っています。ここに原因となるライブラリ固有の詳細な情報（例：`IO Error: Permission denied`）を直接書くと、メッセージが複雑化しすぎてしまいます。

#### 2. `Source`トレイトの責務：「デバッガへの証拠責任」
一方、エラー型は通常、`std::error::Error`トレイトを実装します。そして最も重要なのが、原因となった別のエラー型を保持するための機構です。これが**原因（Cause）を指し示すリンク**として機能するのが事実上の`source()`の役割です。これは「なぜ失敗したのか」という技術的な根拠であり、「何が起きたか」の説明ではありません。

#### 3. `Reporter`ライブラリの責務：「情報の整形・統合責任」
`anyhow`や`eyre`といった外部クレートは、このエラー型を渡されて「ログとして美しく表示する」役割（**Reporter/Wrapper**）を果たしています。これらのラッパーが頑張りすぎて、「Displayの内容」と「Sourceから辿った原因のDisplay内容」の両方を出し惜しみなく出力しようとするために、重複が発生してしまうわけです。

この構造を理解すると、エラー型設計は単なるコード記述ではなく、**情報フローにおけるレイヤーごとの責任分界点（Boundary）を設定するアーキテクチャ作業**だと認識できます (￣▽￣)。

### 📊 エラーメッセージ表示の責務分離比較表

| 要素 | 目的/役割 | 対象読者 | 期待される出力 | 重複リスク |
| :--- | :--- | :--- | :--- | :--- |
| **`Display`** | **自己の説明**。高レベルな文脈を添える。 | エンドユーザー、オペレーター | 「処理が失敗しました：認証情報に誤りがあります。」（抽象的） | 低 (自身のみの責任) |
| **`Source`/`#[source]`** | **原因の伝達**。技術的な発生源を示すリンク。 | SRE、バックエンドエンジニア | `std::io::Error: Permission denied`（具体的） | 高 (Reporterが再帰的にたどるため) |
| **`anyhow`などのラッパー** | **情報の統合・整形**。コンテキストの追加。 | デバッガ全体 | 「[レイヤーA]で失敗：認証情報に誤りがあります。（原因：Permission denied）」 | 中 (重複表示の原因となる) |

### 🎨 設計パターン徹底解説：ベストプラクティスによる分離戦略

では、具体的な設計上の判断基準はどうなるのか。ここでは「文脈追加」と「透過ラッパー」の2つの主要なケースに分けて、最も堅牢なパターンを提示します。

#### パターンA: 文脈を追加する場合（自レイヤーが処理ロジックを持つ時）
この場合、エラーは**自分自身でメッセージの一部を定義する責務**を持っています。

*   **目的:** 「どの機能のどの部分で」失敗したのかという情報を追加したい。
*   **推奨パターン:** `#[error("自レイヤが検出した文脈的な説明")]` + `#[source]`
*   **仕組み:** エラー型全体を定義しつつ、フィールドとして原因となるエラーを受け取るようにします。これにより、「このメッセージは私（ラッパー）が追加した情報だ」という明確な境界線を引きながら、元のエラーの原因を示すことができます。

#### パターンB: 単純に原因をラップする場合（単なる通過点の場合）
これは最も陥りやすい罠です。ただ単に「前のレイヤーのエラーを受け取って、それをそのまま渡すだけ」のケースです。

*   **目的:** エラーの型定義を簡潔にしつつ、内部で別のエラー型を保持したい。
*   **推奨パターン:** `#[error(transparent)]`
*   **仕組み:** `#[error(transparent)]`を使うと、自分自身のフィールドやメッセージ定義を無視し、「中身（inner type）」が持つ`Display`情報と`Source`情報をそのまま引き継ぐようにRustコンパイラに強制できます。これにより、メッセージの重複によるノイズを最小限に抑えられます。

#### 🚨 実践コード例：パターンB（透過ラップ）の実装
単純な認証レイヤーで、DB接続エラーを受け取る場合の理想的な実装を見てみましょう。

```rust
use std::error::Error;
use thiserror::Error;

// DBクライアントから返された具体的なIOエラーを単に保持するラッパー
#[derive(Debug, Error)]
pub struct DbAccessLayerError {
    // Inner typeとして元のエラーを受け取る
    inner: Box<dyn Error + Send + Sync>, 
}

// #[error(transparent)] を使うことで、この構造体自体のDisplay定義を最小限に抑える。
#[derive(Debug, Error)]
#[error(transparent)] // これが肝！内部の型の表示ロジックをそのまま利用する
pub struct DatabaseError(#[source] Box<dyn Error + Send + Sync>);

impl From<std::io::Error> for DatabaseError {
    fn from(e: std::io::Error) -> Self {
        DatabaseError(Box::new(e))
    }
}

// use anyhow; // anyhowもこの仕組みを利用している

fn main() {
    // 例として、何らかのIOエラーが発生したとする
    let original_error = std::io::ErrorKind::NotFound.into(); 
    let db_err: DatabaseError = DatabaseError(Box::new(original_error));

    println!("--- Display表示（ユーザー向け） ---");
    // この場合、#[error(transparent)]のおかげで، innerのSource情報が重複せずに綺麗に出る。
    println!("{}", db_err); 

    println!("\n--- Source辿り（デバッグログ用） ---");
    // source()も内部のエラーを適切に指し示す
    if let Some(source) = db_err.source() {
        eprintln!("Source detected: {}", source);
    }
}
```

#### 🚨 実践コード例：パターンA（文脈追加）の実装
ログイン処理レイヤーで、認証失敗の文脈を付け加えたい場合。

```rust
use std::error::Error;
use thiserror::Error;

// 独自のカスタムエラー定義を行う
#[derive(Debug, Error)]
pub enum AuthFlowError {
    // #[source] をつけることで、このエラーが原因のエラーを受け入れられるようにする
    #[error("認証処理中に問題が発生しました：{0}")]
    AuthenticationFailed(#[source] Box<dyn Error + Send + Sync>),

    // 自分で判断したビジネスロジック上のエラー（原因は外部に依存しない）
    #[error("ユーザーIDが存在しません。'{0}'を確認してください")]
    InvalidUserId(String),
}

fn perform_auth(user_id: &str) -> Result<(), AuthFlowError> {
    if user_id.is_empty() {
        return Err(AuthFlowError::InvalidUserId("ユーザーID".to_string()));
    }
    let dummy_io_error = std::io::ErrorKind::NotFound.into(); 
    // DBエラーを受け取り、認証フローの文脈でラップする
    Err(AuthFlowError::AuthenticationFailed(Box::new(dummy_io_error)))
}

fn main() {
    // ... (実行例)
}
```

### ⚖️ 実践への示唆：レイヤー設計におけるエラーハンドリング戦略

ここまで技術的な深掘りをしてきましたが、最終的にエンジニアとして知っておくべきは、「どのパターンを選ぶか」という判断軸です。これは単なるコーディングテクニックではなく、**システム全体のアーキテクチャ設計思想に関わる問題**になります。

#### 1. 「どこでエラーメッセージを定義するか？」の原則
*   **ビジネスロジックレイヤー (Service Layer):** ここが「何が間違っているか」という**文脈（Context）**を最もよく知っています。「ユーザーIDが存在しない」「権限がない」といった、システム固有の概念的なエラーはここで発生し、メッセージも定義すべきです。
*   **データアクセスレイヤー (Repository/DB Layer):** ここで扱うのは、あくまで「外部との通信失敗」という技術的なエラー（`std::io::Error`, `sqlx::Error`など）です。これらのエラーをラップする際は、できる限り`#[error(transparent)]`を用いて**透過的に原因を渡す**べきです。メッセージを追加すると、文脈が混ざり、デバッグ時のノイズになります。
*   **API境界レイヤー (Controller/Handler):** 最終的な呼び出し元で、すべてのエラーを集約し、「ユーザーに何を伝えるか」という視点（Display）と「ログに残すべき技術情報」（Source）を分離して整形するのが理想です。

#### 2. `anyhow` vs `thiserror`の使い分け
この設計指針は、どのクレートを使うかにも影響します。

*   **`thiserror`:** **推奨度：高**。自前のエラー型（カスタムエラー）を定義し、厳密な責務分担を行う必要がある場合に必須です。メッセージと原因を完全に分離できます。
*   **`anyhow`:** **推奨度：中〜高**。特にプロトタイプやCLIツールなど、「とにかく動けばいい」「デバッグ時に全体像を見たい」という初期段階で非常に強力ですが、大規模なライブラリ設計の際は、過信しすぎるとメッセージが肥大化しやすい点に注意が必要です。

### 🚀 まとめ：次にやるべきアクションと知識の定着化

エラーハンドリングは、「バグを避けるためのコーディングテクニック」ではなく、**「システムの信頼性を保証するためのアーキテクチャ原則」**だと捉え直すことが極めて重要です。

もしあなたが今、大規模なバックエンドAPIを設計しているなら、すぐに以下のチェックリストを自分自身に課してください：

1.  **エラー型の定義:** どのレイヤーでどのカスタムエラー型を定義しているか？
2.  **ラップの判断:** そのエラーは「単なる原因の通過点」なのか、「独自の文脈を持つべきロジック失敗」なのか？
3.  **表示テスト:** `anyhow::Context`を通す際、ログ出力が意図せずメッセージを重複させていないか？

この責務分離の視点を意識し、`transparent`なラップとカスタムコンテキスト追加の境界線を明確にすることで、あなたのシステムは「エラーが出た時こそ美しい」という、真の意味で堅牢な設計を手に入れることができるはずです。マジで試してみてほしい！(^_^)
